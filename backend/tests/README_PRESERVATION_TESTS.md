# Preservation Property Tests

## Overview

The preservation property tests in `test_preservation_properties.py` capture baseline behavior that must be preserved after bugfixes are applied. These tests follow the **observation-first methodology** - they are designed to PASS on the UNFIXED code to confirm the baseline behavior.

## Purpose

These tests verify that non-buggy inputs continue to produce the same behavior after fixes are applied. They cover:

1. **Model Routing** (Requirements 3.1, 3.2)
   - Qwen 2.5:7b remains default model for standard decisions
   - DeepSeek-R1:7b continues to be invoked for HIGH/CRITICAL severity anomalies

2. **Approval Gates** (Requirements 3.3, 3.4)
   - Actions with cost impact below ₹50,000 continue to auto-execute without approval
   - Actions with cost impact above ₹50,000 continue to route to approval queue

3. **Demo Reset Behavior** (Requirement 3.5)
   - Demo reset continues to re-seed 450 transactions, 200 licenses, 50 SLA tickets

4. **API Schemas** (Requirements 3.6, 3.7, 3.8, 3.9)
   - GET /api/savings/summary returns all 9 required fields
   - GET /api/anomalies/ returns all required fields
   - GET /api/actions/ returns all required fields
   - GET /api/audit/ returns all required fields

5. **Database Schema** (Requirements 3.10, 3.11, 3.12)
   - All 8 required tables exist (vendors, transactions, licenses, sla_metrics, anomaly_logs, actions_taken, approval_queue, audit_trail)
   - sla_deadline column continues to be auto-generated by PostgreSQL
   - actions_taken table supports filtering by status, action_type, date ranges

6. **Frontend Behavior** (Requirement 3.13)
   - Dashboard continues to poll /api/savings/summary every 10 seconds

7. **Model Management** (Requirements 3.17, 3.18, 3.19, 3.20)
   - DeepSeek unloads current model before loading, reloads Qwen after completion
   - DeepSeek hourly budget (10 calls) continues to be enforced
   - Llama 3.2:3b continues to be used as fallback

## Prerequisites

Before running these tests, ensure the following services are running:

1. **PostgreSQL Database**
   ```bash
   docker compose up -d postgres
   ```

2. **Redis**
   ```bash
   docker compose up -d redis
   ```

3. **Backend API Server**
   ```bash
   docker compose up -d backend
   # OR run locally:
   cd cost-intelligence/backend
   python main.py
   ```

4. **Ollama (for model tests)**
   ```bash
   docker compose up -d ollama
   ```

## Running the Tests

### Run all preservation tests:
```bash
cd cost-intelligence/backend
python -m pytest tests/test_preservation_properties.py -v
```

### Run specific test categories:
```bash
# Model routing tests
python -m pytest tests/test_preservation_properties.py -k "model_routing" -v

# API schema tests
python -m pytest tests/test_preservation_properties.py -k "api_schema" -v

# Database schema tests
python -m pytest tests/test_preservation_properties.py -k "database_schema" -v

# Property-based tests
python -m pytest tests/test_preservation_properties.py -k "property" -v
```

### Run with property-based testing details:
```bash
python -m pytest tests/test_preservation_properties.py -v --hypothesis-show-statistics
```

## Expected Outcome

**CRITICAL**: These tests MUST PASS on the UNFIXED code.

If these tests pass on unfixed code, they confirm the baseline behavior that must be preserved. After applying bugfixes, these same tests should still pass, proving that non-buggy functionality was not affected by the fixes.

## Test Structure

### Unit Tests
- Test specific preservation requirements (e.g., approval threshold, model configuration)
- Verify API response schemas
- Check database schema structure

### Property-Based Tests
- Generate random inputs across the valid input domain
- Verify model routing logic is preserved for all severity/cost combinations
- Verify anomaly detection logic is preserved for all anomaly types
- Provide stronger guarantees than unit tests by testing many cases automatically

## Integration with Bug Condition Tests

The preservation tests complement the bug condition exploration tests:

1. **Bug Condition Tests** (`test_bug_condition_exploration.py`)
   - Test inputs that TRIGGER the bugs
   - EXPECTED TO FAIL on unfixed code
   - EXPECTED TO PASS after fixes are applied

2. **Preservation Tests** (`test_preservation_properties.py`)
   - Test inputs that DO NOT trigger the bugs
   - EXPECTED TO PASS on unfixed code
   - EXPECTED TO PASS after fixes are applied (no regressions)

Together, these tests provide comprehensive coverage:
- Bug condition tests verify fixes work correctly
- Preservation tests verify fixes don't break existing functionality

## Troubleshooting

### Database connection errors
```
asyncpg.exceptions.InvalidCatalogNameError: database "cost_intelligence" does not exist
```
**Solution**: Start the PostgreSQL container and ensure the database is initialized:
```bash
docker compose up -d postgres
docker compose exec postgres psql -U ci_user -d cost_intelligence -c "\dt"
```

### HTTP connection errors
```
httpx.ConnectError: All connection attempts failed
```
**Solution**: Start the backend API server:
```bash
docker compose up -d backend
# Verify it's running:
curl http://localhost:8000/api/system/status
```

### Property-based test failures
If property-based tests find counterexamples, review the generated inputs in the test output. Hypothesis will show the specific values that caused the failure.

## Maintenance

When adding new preservation requirements:

1. Add a new test function following the naming convention: `test_preservation_<feature>_<behavior>`
2. Document which requirements it validates using the docstring format
3. Ensure the test passes on unfixed code before committing
4. Add the test to the appropriate category in this README
